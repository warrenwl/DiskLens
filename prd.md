# DiskLens PRD

## 1. 文档信息

- 产品名称：DiskLens
- 文档类型：产品需求文档（PRD）
- 当前阶段：MVP 持续迭代
- 适用平台：macOS 13+
- 关联文档：
  - `README.md`
  - `docs/product-design.md`
  - `docs/iteration-plan.md`

## 2. 项目背景

AI 与开发工作流会在本地持续累积模型文件、构建产物、缓存、依赖目录和卸载残留。用户通常知道“磁盘快满了”，但很难快速回答三个问题：

1. 空间到底被谁占用了。
2. 哪些内容可以安全处理。
3. 怎样在不误删重要数据的前提下完成清理。

现有通用清理工具常见两个问题：

- 可视化不足，只告诉用户“哪里大”，却不能建立空间分布的整体认知。
- 自动化过强，容易把缓存、应用数据、模型文件和用户数据混在一起，降低信任感。

DiskLens 的定位不是“激进清理器”，而是“面向 AI / 开发用户的磁盘可视化与审慎清理工作台”。

## 3. 产品愿景

让用户在几分钟内完成从“看清空间结构”到“确认并清理安全项”的闭环，同时始终保留对风险和操作结果的理解与控制。

核心原则：

- 先看清，再操作。
- 先建议，再执行。
- 默认安全，避免误删。
- 只移入废纸篓，不做永久删除。

## 4. 目标用户

### 4.1 核心用户

- 本地运行 AI 模型、ComfyUI、Ollama、HuggingFace / ModelScope 生态的用户
- 频繁使用 Node.js、Python、Docker、Homebrew、Xcode 等开发工具的开发者
- 磁盘空间经常紧张、但不愿使用“黑盒清理器”的 macOS 用户

### 4.2 用户特征

- 知道磁盘空间被大量工具链和缓存吞噬，但不一定清楚具体路径
- 对“应用数据”“容器数据”“模型文件”有风险意识，不愿盲删
- 需要可解释、可复核、可回退的清理流程

## 5. 用户问题与场景

### 5.1 关键问题

- 模型、缓存、构建产物、重复文件分散在多个目录，缺少统一视图
- Finder 难以从全局维度识别大户目录和清理优先级
- 用户知道可以删缓存，但不知道哪些目录安全、哪些目录危险
- 想清理应用卸载残留，但又担心误删仍在使用的数据

### 5.2 典型场景

1. 用户发现磁盘只剩几十 GB，希望快速定位最大占用目录。
2. 用户需要清理 ComfyUI、npm、uv、pip、Docker、构建缓存等可重建内容。
3. 用户想检查卸载应用留下的偏好、缓存、日志等残留内容。
4. 用户想找出重复文件和超大文件，但不希望系统默认替他做决定。
5. 用户需要导出一份清理报告，便于留档、复盘或分享。

## 6. 产品目标

### 6.1 业务目标

- 建立一个面向 AI / 开发场景的差异化磁盘管理工具
- 以“可视化 + 风险分级 + 只移入废纸篓”建立用户信任
- 为后续更强的清理能力预留产品与信息架构基础

### 6.2 用户目标

- 在 1 次扫描后看清主要空间占用结构
- 快速识别高价值清理候选项
- 在不做永久删除的情况下完成一轮安全清理

### 6.3 当前版本目标

- 提供首页、磁盘全景、一键清理三段式主流程
- 支持只读扫描、可视化浏览、风险分级、建议命令、报告导出
- 支持在用户确认后，将已选安全候选项移入废纸篓

## 7. 非目标

当前版本明确不做以下能力：

- 永久删除文件
- 自动执行 Docker prune、模型删除、包管理器清理命令
- 默认清理 Application Support / Containers / Group Containers 等高风险路径
- root helper、后台提权、系统级守护进程
- 云同步、跨设备对比、磁盘健康硬件诊断

## 8. 产品范围

### 8.1 当前已具备/已落地基础

基于当前仓库代码，产品已经具备以下基础能力：

- 首页、磁盘全景页、一键清理页三种主界面状态
- 扫描默认重点目录、全 Data 卷、手动选择目录
- Treemap 磁盘占用可视化、下钻、面包屑导航
- 风险分级、路径搜索、大小阈值筛选、排序
- 历史扫描缓存与历史记录展示
- Markdown / JSON / SVG / PNG 导出
- 重复文件检测
- 卸载残留扫描
- 一键清理候选项汇总与勾选
- 将已选项目移入废纸篓，并显示完成结果

### 8.2 本 PRD 覆盖范围

本 PRD 重点约束以下产品能力：

- 入口与页面结构
- 扫描与全景浏览体验
- 清理候选项生成规则
- 风险控制与执行边界
- 输出物、指标与验收标准

## 9. 核心价值主张

DiskLens 为用户提供三层价值：

1. 看见：用 Treemap 和结构化列表快速看清空间分布。
2. 判断：用分类、风险等级、建议动作帮助用户建立信心。
3. 执行：把用户确认后的安全候选项移入废纸篓，而非直接删除。

## 10. 用户流程

### 10.1 主流程 A：磁盘全景

首页 -> 点击“磁盘全景” -> 发起扫描 -> 查看 Treemap -> 下钻目录 -> 查看详情/建议 -> 搜索筛选 -> 导出报告

### 10.2 主流程 B：一键清理

首页 -> 点击“一键清理” -> 准备扫描结果 + 卸载残留 + 重复文件 -> 生成候选分组 -> 用户调整勾选 -> 点击“一键清理” -> 二次确认 -> 移入废纸篓 -> 查看结果摘要

### 10.3 辅助流程

- 首页查看最近一次扫描状态
- 从权限提示进入“完全磁盘访问”
- 打开历史扫描记录进行参考
- 导出结果用于复核或外部沟通

## 11. 功能需求

### 11.1 首页

目标：用最小选择成本引导用户进入两条核心路径。

需求：

- 首屏默认进入首页，不自动开始扫描
- 显示产品名、状态摘要、上次扫描时间（若存在缓存）
- 提供两个主按钮：
  - 磁盘全景
  - 一键清理
- 首页需传达三条信任信息：
  - 只移入废纸篓
  - 默认保护用户数据
  - 支持可视化定位

验收标准：

- 应用启动后不自动扫描
- 用户可以从首页直接进入磁盘全景或一键清理
- 首页可展示已有历史扫描痕迹

### 11.2 磁盘扫描

目标：生成可用于全景展示与清理分析的扫描结果。

需求：

- 支持三种扫描模式：
  - 重点目录
  - 全 Data 卷
  - 手动选择目录
- 扫描过程中展示：
  - 当前阶段
  - 当前路径
  - 已扫描目录数 / 文件数
  - 已累计字节数
  - 不可读路径摘要
- 支持取消扫描
- 扫描结束后缓存结果，并写入历史

验收标准：

- 用户可在扫描中感知进度
- 无权限目录不会导致应用崩溃
- 扫描结果可被后续导出、全景展示与清理流程复用

### 11.3 磁盘全景（Panorama）

目标：让用户建立“空间是怎么被占满的”全局认知。

需求：

- 使用 Treemap 展示当前扫描范围内的空间占用
- 支持点击下钻与面包屑返回
- 支持 hover / 选中查看详情
- 右侧展示当前选中项的说明和清理建议
- 下方表格支持：
  - 搜索
  - 风险筛选
  - 大小阈值筛选
  - 排序
  - 复制路径
  - Finder 中显示
  - 复制建议命令

验收标准：

- 用户能够从 Treemap 快速进入大目录并回退
- 表格筛选与 Treemap 内容可配合使用
- 建议动作必须是可解释的，不能只有“建议清理”四个字

### 11.4 清理候选项准备

目标：生成可复核、可勾选、默认安全的清理计划。

需求：

- 一键清理页打开后自动准备清理计划
- 若已有扫描结果，优先复用；否则先发起扫描
- 并行/连续生成以下候选项来源：
  - 安全清理项
  - 卸载残留
  - 重复文件
  - 大文件
- 明确展示“准备中”状态

验收标准：

- 冷启动进入一键清理时可自动完成准备
- 已有扫描结果时无需重复全量扫描
- 准备完成后能展示四个固定分组

### 11.5 清理分组与默认选择

目标：让默认选择体现“保守、安全、可解释”。

分组规则：

1. 安全清理
2. 卸载残留
3. 重复文件
4. 大文件

默认选择规则：

- 安全清理：默认选中
- 卸载残留：仅默认选中安全残留
- 重复文件：默认不选
- 大文件：默认不选

候选项行信息要求：

- 勾选框
- 名称
- 路径
- 大小
- 风险等级
- 原因 / 建议说明

验收标准：

- 每个分组可单独全选/全不选
- 总选中数量和总选中字节数实时更新
- 重复文件默认保留每组第一个文件
- 大文件不会被自动勾选

### 11.6 清理执行

目标：在用户确认后，以最保守方式执行清理。

需求：

- 顶部提供“一键清理”执行按钮
- 当没有已选候选项时按钮不可执行
- 点击执行前必须展示确认弹窗，明确说明：
  - 将处理的项目数
  - 预计释放空间
  - 只会移入废纸篓，不会永久删除
- 执行方式仅允许 `trashItem`
- 执行后展示结果摘要：
  - 成功项数
  - 预计释放空间
  - 失败项数
- 成功移入废纸篓的候选项需从当前列表移除

验收标准：

- 不允许永久删除
- 不允许绕过确认弹窗
- 执行失败的项目仍然保留在结果中供用户复查

### 11.7 报告导出与历史

目标：让结果可保存、可比较、可复用。

需求：

- 支持导出：
  - Markdown
  - JSON
  - SVG
  - PNG
- 历史记录支持查看过往扫描摘要
- 历史记录至少应包含：
  - 扫描时间
  - 扫描模式
  - 扫描字节数
  - 不可读路径数量
  - 扫描耗时

验收标准：

- 导出文件可成功生成
- 历史记录不会因为新扫描而全部丢失
- Markdown 报告可用于人工复核

### 11.8 权限与安全提示

目标：在不自动提权的前提下清楚提示权限边界。

需求：

- 当扫描到不可读路径时，需在界面中保留摘要信息
- 提供进入“完全磁盘访问”的引导入口
- 明确说明应用不使用 root helper、不自动提权

验收标准：

- 用户即使没有完全磁盘访问，也能完成部分扫描并获得反馈
- 权限不足时应用行为应可预测、可解释

## 12. 分类与风险策略

### 12.1 重点识别对象

- AI 模型与推理资源
- 各类缓存目录
- 开发构建产物
- Docker 数据
- 应用程序及卸载残留
- 用户数据
- 敏感/应用数据
- 系统相关数据

### 12.2 风险等级

- 可安全清理：可重建缓存、低风险临时产物
- 谨慎清理：可能有用，但需要人工判断
- 建议保留：大概率仍有业务价值或可能影响使用
- 系统相关：默认不应作为清理目标

### 12.3 风险策略原则

- 风险信息必须跟随候选项展示
- 默认勾选只能覆盖低风险内容
- 与用户数据、系统路径、容器数据强相关的内容不能默认勾选

## 13. 非功能需求

### 13.1 性能

- 扫描时界面需持续响应，不阻塞主线程
- 默认扫描应优先覆盖用户最常见的高占用目录
- 大目录场景下需提供进度反馈与取消能力

### 13.2 安全

- 默认只读扫描
- 所有清理动作仅限移入废纸篓
- 不后台执行用户不可见的 destructive 操作

### 13.3 可解释性

- 每个风险等级和建议动作都应可被理解
- 清理前、清理后都要有明确状态和结果反馈

### 13.4 可靠性

- 权限不足、文件被占用、路径失效等情况不得导致程序崩溃
- 失败结果必须可追踪、可复查

## 14. 成功指标

### 14.1 体验指标

- 用户能在一次扫描后快速识别前几大空间占用来源
- 用户能理解默认选中项为何安全
- 用户能在一次清理后明确知道哪些项目被移入废纸篓、哪些失败

### 14.2 产品指标（建议后续埋点）

- 首页到首次扫描启动转化率
- 首页到一键清理启动转化率
- 清理准备完成率
- 清理确认率
- 清理执行成功率
- 平均扫描耗时
- 平均已选清理空间

## 15. 页面级规格

### 15.1 信息架构

当前版本信息架构保持三层主导航，不引入复杂侧边栏或多级菜单：

1. 首页
2. 磁盘全景
3. 一键清理

页面跳转规则：

- 应用启动后进入首页
- 首页可进入磁盘全景或一键清理
- 磁盘全景和一键清理都必须支持返回首页
- 当前版本不引入独立设置页、偏好页或多窗口复杂导航

### 15.2 首页规格

页面目标：建立产品信任感，并让用户一眼理解两条主路径。

页面结构：

- 顶部：Logo、产品名、最近扫描时间
- 左侧主视觉区：
  - 健康状态视觉符号
  - 核心价值文案
  - 三条信任标签
  - 磁盘使用概览
- 右侧动作区：
  - 磁盘全景按钮
  - 一键清理按钮
  - 权限/历史状态提示

交互要求：

- 点击“磁盘全景”后进入全景页；若当前无扫描结果，则自动开始扫描
- 点击“一键清理”后进入清理页并开始准备清理计划
- 若有历史扫描缓存，应在首页给出轻量状态提示，而不是打断用户

页面状态：

- 无缓存状态
- 有缓存状态
- 存在不可读路径提示状态

### 15.3 磁盘全景页规格

页面目标：让用户在空间认知和清理判断之间自然切换，但不在此页直接混入清理主流程。

页面结构：

- 顶部工具栏：
  - 返回首页
  - 扫描模式切换
  - 手动选择目录
  - 开始扫描 / 停止扫描
  - 导出菜单
  - 历史按钮
- 主体区域：
  - 左侧 Treemap 可视化
  - 右侧详情面板
- 底部/右下列表区：
  - 文件表格
  - 搜索、筛选、排序辅助

详情面板内容：

- 当前选中项名称
- 路径
- 占用大小
- 分类
- 风险等级
- 推荐操作
- 相关建议列表

关键交互：

- 点击 Treemap 节点可选中
- 双击或进入动作可下钻
- 通过面包屑返回上级层级
- 表格与 Treemap 共享同一份扫描结果语义
- 导出应基于当前扫描结果，而非当前视觉筛选后的局部结果

页面状态：

- 空状态：尚未扫描
- 扫描中状态
- 扫描完成状态
- 扫描取消状态
- 存在不可读路径状态

### 15.4 一键清理页规格

页面目标：把“生成候选项 -> 人工复核 -> 确认执行”做成一个明确、可信赖的闭环。

页面结构：

- 顶部工具栏：
  - 返回首页
  - 当前已选数量与空间
  - 重新检索
  - 一键清理执行按钮
- 主内容区：
  - 准备中视图，或
  - 清理结果 Banner，或
  - 四个清理分组列表

四个清理分组：

1. 安全清理
2. 卸载残留
3. 重复文件
4. 大文件

分组组件要求：

- 标题
- 副标题/规则说明
- 分组级勾选控制
- 分组内候选项列表
- 每行候选项支持：
  - 勾选
  - 查看路径
  - Reveal in Finder
  - 复制路径

执行前逻辑：

- 仅当至少有 1 个候选项被选中时，执行按钮可用
- 用户点击执行后必须弹出确认框
- 确认框应再次强调“仅移入废纸篓”

执行后逻辑：

- 成功项从当前候选列表移除
- 失败项保留并可继续查看
- Banner 展示成功/失败摘要

页面状态：

- 初次进入准备中
- 准备完成但无候选项
- 准备完成且存在候选项
- 执行中
- 执行完成
- 执行部分失败

### 15.5 全局状态与文案原则

所有关键状态都需要有明确用户可感知反馈：

- 扫描中：告诉用户系统正在做什么
- 检索中：告诉用户正在生成哪些候选项
- 权限不足：告诉用户为什么结果可能不完整
- 清理执行中：告诉用户当前动作不可视为“删除”
- 清理完成：告诉用户处理了什么、失败了什么

文案要求：

- 尽量避免“清理成功释放 X GB”这类带强承诺的说法
- 优先使用“预计释放”“已移入废纸篓”等可验证表达
- 所有高风险项都应强调“需确认用途”

## 16. 版本规划与交付范围

### 16.1 V0.1 可发布版本

目标：形成完整的“首页 -> 全景 -> 一键清理”主流程。

必须具备：

- 首页入口结构稳定
- 扫描流程稳定
- Treemap 可浏览、可筛选、可导出
- 一键清理可生成候选项并移入废纸篓

不要求：

- 特别丰富的分类覆盖率
- 高级历史分析
- 自动化命令执行

### 16.2 V0.2 增强版本

目标：提升规则命中率与结果解释力。

建议增强：

- 扩大 AI / 开发工具链规则覆盖
- 优化重复文件分组展示
- 优化大文件说明文案
- 增加历史趋势对比
- 增强异常与失败反馈

### 16.3 V0.3 进阶版本

目标：在不破坏信任边界的前提下增强“建议能力”。

可考虑方向：

- 更多结构化建议命令
- 扫描变化对比
- 更丰富的清理前预估与解释
- 更强的筛选、聚合和定位能力

## 18. 验收场景

### 18.1 首页验收

- 冷启动进入首页
- 首页无自动扫描
- 历史扫描存在时能展示最近状态

### 18.2 磁盘全景验收

- 用户可切换扫描模式并开始扫描
- 扫描中可看到进度与当前路径
- 扫描完成后可查看 Treemap、详情和表格
- 用户可导出报告并打开历史记录

### 18.3 一键清理验收

- 冷启动进入清理页可自动完成准备
- 四类清理分组能正常展示
- 默认选择规则符合预期
- 用户可修改勾选并发起执行
- 执行后只移入废纸篓，不永久删除

### 18.4 异常验收

- 无权限目录不崩溃
- 文件不存在或被占用时能显示失败结果
- 无候选项时有空状态提示
- 取消扫描或取消检索时状态明确

### Milestone 2：安全清理闭环

- 失败项和结果摘要清晰

### Milestone 3：规则与体验增强

## 20. 风险与待决事项

### 16.1 风险

- 部分目录受 macOS 权限限制，扫描结果可能不完整
- “安全清理”的规则命中若过于宽松，会损害用户信任
- 大文件与重复文件虽然技术上可识别，但业务价值高度依赖用户判断

### 16.2 待决事项

- 后续是否开放对更多工具链缓存的结构化识别
- 是否在未来版本支持“推荐命令执行”而非仅复制
- 是否提供更强的历史对比和空间变化趋势视图
- 是否将清理结果与扫描历史做关联留档

## 21. 验收清单

- 应用启动进入首页，不自动扫描
- 磁盘全景流程可完整运行
- 一键清理可在冷启动和已有扫描两种情况下完成准备
- 四类候选分组都能展示
- 默认选择仅覆盖安全清理与安全残留
- 清理前必须确认，清理后只移入废纸篓
- 导出 Markdown / JSON / SVG / PNG 正常
- 历史记录可查看
- 权限不足时有明确提示且不崩溃

## 22. 结论

DiskLens 当前最重要的产品取舍不是“能清多少”，而是“用户是否愿意相信它”。因此本 PRD 将产品核心定义为：

- 以 Treemap 建立空间认知
- 以分类与风险分级建立判断依据
- 以废纸篓式执行建立操作安全感

只要这三件事持续成立，DiskLens 就能在 AI / 开发磁盘管理场景中形成稳定而清晰的产品价值。
